# P0.3 Conditional Stage 1 Preregistration Plan

Generated: 2026-07-07

This plan is conditional on P0.3 nonlinear-ground-truth closure passing local
CPU verification. It does not start GPU work and does not modify the KBS
manuscript.

## Development And Confirmatory Split

- `data_seed=0` is development only.
- `data_seed=0` must not enter Stage 1 confirmatory aggregation,
  statistical testing, or manuscript performance claims.
- P0.3 may inspect trajectories at iterations `120`, `500`, and `2000` on
  development seed only.
- After P0.3, one shared primary checkpoint iteration must be frozen before any
  confirmatory Stage 1 results are viewed.
- All methods use the same frozen primary checkpoint iteration.
- Other checkpoint iterations are retained only as trajectory diagnostics.
- No checkpoint may be selected using graph-recovery performance on
  confirmatory seeds.
- If a validation rule is used instead of a fixed checkpoint, it must be frozen
  before Stage 1 and must not use graph ground truth.

## Formal Methods

Formal Stage 1 methods:

- `RawTargetBaselineJRNGC`
- `CPDepthwiseISTFJRNGC`
- `FixedResidualFIR3JRNGC`
- `FixedEMAJRNGC`

Limited ablation:

- `RawChainMambaISTFJRNGC`

The Mamba candidate remains diagnostic only unless its temporal and truncation
semantics pass the same gates as the formal nominal-lag methods.

## Factorial Cells

All methods are evaluated on D2 cells generated by
`factorial_data.generate_factorial_cell`:

- `Stat+Linear`
- `Stat+Nonlinear`
- `NS+Linear`
- `NS+Nonlinear`

The P0.3 generator uses fixed coordinate-wise nonlinearity
`g_i(p_i) = (1-beta)p_i + beta*s0*tanh(p_i/s0)` with pre-frozen
`beta=0.5`, `s0=0.5` under D2.

## Stage 1a

- `data_seeds = [1, 2, 3]`
- `train_seeds = [0, 1]`
- all four factorial cells
- all four formal methods
- limited Mamba ablation on a reduced budget only if local P0.3 diagnostics
  show it is computationally and semantically usable as a diagnostic.

## Stage 1b

Stage 1b is run only if Stage 1a gates pass:

- `data_seeds = [1, 2, 3, 4, 5]`
- `train_seeds = [0, 1]`
- all four factorial cells
- all four formal methods

## Score Reporting Tracks

Nominal-lag primary track:

- baseline
- CP-depthwise
- FixedFIR3 only if temporal-mass and truncation gates pass

Full-H any-lag reference track:

- baseline
- CP-depthwise
- FixedFIR3
- EMA
- limited Mamba

EMA must use `H=64` or full-prefix attribution if `H=32` omitted mass remains
above `0.01`. Truncated `H=32` EMA scores must not be described as complete
EMA attribution.

Formal graph metrics use full eligible-window exact extraction with chunking
for memory control. Window counts such as 32, 64, and 128 are sensitivity
audits, not main-table results.

## Metric Adapter

- Score convention is `score[target, source]`.
- Ground truth for summary metrics is any-lag graph support.
- AUROC, AUPRC, and threshold F1 use the frozen canonical
  `compute_metrics_multimode(...)[summary_max]` path.
- Exact-top-k F1, MCC, SHD, and nSHD use `src.knowledge_metrics`.
- The formal adapter order is `compute_metrics(gc_true, gc_pred)`.

## Go/No-Go Gates

Rewriting the KBS main method around CP-depthwise is allowed only if all gates
below pass:

- CP-depthwise improves over baseline in at least two confirmatory cells.
- At least one improved cell is non-stationary.
- Paired mean AUROC improvement is approximately `>= 0.03`.
- AUPRC and MCC do not show simultaneous material degradation.
- The result is not driven by a single data seed.
- CP-depthwise is not fully matched or dominated by FixedFIR3.
- CP-depthwise is not fully dominated by EMA on the full-H reference track.
- Cross-variable leakage remains numerically zero.
- Temporal-horizon and truncation gates continue to pass.
- Learned kernel norm, identity deviation, and filtered/raw variance ratio show
  nontrivial but controlled filter learning.

If CP-depthwise is effectively equivalent to baseline or FixedFIR3, the project
should stop treating it as a KBS main-method result and pivot to either a
mechanism/audit paper or a redesigned method.
